## 1. One-paragraph summary of reality

Even Keel Learning is a **Next.js 14 / React 18 / TypeScript / Tailwind** prototype. Eight
routes render end-to-end with polished UI, 9 translated locales (including RTL
Arabic), a working Socratic chat engine that refuses direct answers, a working
keystroke-cadence mimicry analyser, a working comprehension gate, a working
mock regulatory-absorb engine, and a working offline sync scaffold. It runs
entirely in the browser. There is **no back-end, no database, no LLM inference,
no passkey ceremony, no cross-device sync, and no real cryptographic
signatures** tied to real student identity. Every such capability is either a
deterministic mock (clearly marked below) or a client-side demo that is safe to
show investors but not safe to deploy to schools.

### 4.2 Known, not fixed

- **Subject labels are not translated.** The `SUBJECTS` array in `SubjectGrid.tsx` has English/native labels baked in. "Maths" does not flip to "Matemáticas" in Spanish. Defensible (users expect e.g. "Gaeilge" to show that label regardless of UI language) but should be a product decision.
- **The subject picker is a UI placeholder for content that does not yet exist behind most tiles.** `SubjectGrid.tsx` advertises **64 subjects** across 9 groups (Core, Sciences, Humanities, Languages, Business, Tech, Arts, Life, Vocational). Selecting any tile updates the title label (`"WELDING · today's problem"`), but the actual problem body on `/student` is **hard-coded to a single Maths equation** (`app/student/page.tsx:93` → `"Solve for x:  2x + 5 = 17"`). The deterministic answer-checker (`lib/validation/answer-checker.ts`) is **numeric-only** with no subject branching. The parallel-problem corpus (`lib/eke/parallel-problems.ts`) currently has **one populated skill family** (`linear-eq-1var`, 4 entries). The Comprehension Gate ships three hard-coded linear-equations questions by default. **In short: clicking a non-Maths tile in the picker today does not change the engine's content.** Honouring the picker requires, per subject, (a) a validator or an explicit "qualitative — no auto-check" mode, (b) a hand-written parallel-problem corpus, and (c) Comprehension Gate questions. The picker stays in the UI because the *shape* of "pick your subject" is genuinely useful and because removing 60 tiles would be a worse user experience than the honest disclosure here. A pilot proposal aimed at a non-Maths subject must be explicit that today's automatic content is Maths-only. **v1.5.0 update:** the *pipeline* to close this gap now exists end-to-end (LLM-drafted, teacher-reviewed, signed-manifest delivery). See §4.5 below. v1.5.0 ships ONE migrated and enriched skill family (`maths.linear-eq-1var`); the other 63 tiles still need a teacher to draft and approve content. The pipeline does not write content; it distributes it.
- **RTL audit incomplete.** Arabic renders and reads right-to-left, but a handful of spacing/padding choices use `marginLeft`/`paddingLeft` explicitly. Many are via Tailwind class strings which are fine. Obvious visual flaws should be filed.
- **DSL escalation pipeline is built and hardened; out-of-band provider delivery is honestly stubbed.** v1.4.8 shipped the categorised crisis lexicon (`detectCrisisCategory()`), the signed-and-persisted escalation queue (`lib/safeguarding/escalation-queue.ts`), the school-configurable HTTPS webhook (`lib/safeguarding/webhook-config.ts`), and the Compliance Officer surface. **v1.4.10 adds three hardenings:** (1) deterministic retry-on-schedule with exponential backoff (`lib/safeguarding/retry-scheduler.ts`) so failed deliveries re-attempt automatically while a Compliance dashboard tab is open — browser-only, not a server-side reliable queue; (2) WORM retention (`RETENTION_DAYS = 90`, `pruneExpiredEscalations()`) replacing the v1.4.8 count-based eviction with time-based pruning, signed payloads immutable for the retention period; (3) provider-adapter scaffold (`lib/safeguarding/providers/`) with a real `webhook` adapter and three honest stubs — `email-sendgrid`, `sms-twilio`, `push-fcm` — each returning `kind: "provider_key_required"` with a configHelp string describing what is needed. **What is still not built:** any of the email / SMS / push provider stubs themselves — those each require a paid third-party account, a server-side relay (so the API key never reaches the learner device), and a school billing relationship. Phase 2. The Phase-1 contract is unchanged: HTTPS-only with the school's chosen ingest endpoint. See SAFEGUARDING.md §1.8.
- **Crisis lexicon is English-only and regex-only.** No locale variants and no multi-turn analysis. SAFEGUARDING.md §1 describes the Phase 2 plan.
- **Passkey binding is optional; session-key receipts are still the default for unenroled learners.** v1.4.11 wires WebAuthn so a learner can enrol a device-bound passkey on `/student` and sign receipts with it; those receipts carry `keyType: "passkey-derived"` and the verifier page advertises identity-bound verification. Until a learner actually enrols, the "Sign with session key" button still emits `keyType: "session-demo"` envelopes — which the verifier now labels honestly as "not yet tied to a persistent learner identity". The platform never silently chooses one over the other. Browsers that do not implement WebAuthn render the passkey button as disabled with a "not supported on this browser" caption. **What Phase 1 still does not do:** tie the enroled passkey to a school-issued roster entry (the learner could enrol under arbitrary `learnerInitials`), revoke or rotate credentials server-side, or verify on a second device without re-issuing. Those are Phase-2 items.
- **Transparency bundle is signed with a build-time ephemeral key.** v1.4.9 aggregates governance docs + KCSIE control map + repro manifest + audit manifest into one signed JSON artefact (`evidence/transparency-bundle.json`, copied to `public/transparency-bundle.json`). The signing key is generated, used once, and discarded — the bundle's `signature.keyType: "ephemeral-build-time"` records this. **What this proves:** the four component shas, the `componentDigestB64url`, and the metadata all came from one process at one moment (i.e. the bundle is internally consistent and was not edited after signing). **What this does NOT prove:** that "Even Keel Learning" or any specific institution signed it — no long-lived identity is bound to the key. Phase-2 fix: KMS-backed institution key, or a WebAuthn-passkey-derived signature, replacing the ephemeral key on the build host. Bundle freshness is the verification model in Phase 1: re-build, re-verify, re-publish; `audit:strict` fails CI on any drift. See SAFEGUARDING.md §1.9.

### 4.3 Content authoring pipeline (v1.5.0)

The biggest gap from §4.2 — "subject picker UI shows 64 tiles, only one has real content" — is now closed *in principle* by the v1.5.0 content authoring pipeline. The pipeline is real, signed, and tested. What follows is the honest accounting of what the pipeline does and does not do.

- **No model runs at learner time.** The Eke engine remains a deterministic state machine. The LLM operates only at *authoring time*, off-stage, drafting items into `content/drafts/`. A teacher must rewrite and approve those drafts via `/author` before they reach the signed manifest.
- **Reviewer signing is currently a per-tab session key.** The `/author` UI labels approvals with `"session-demo"` until a reviewer enrols a WebAuthn passkey and the UI is wired to call `signPayload(..., { keySource: "passkey" })`. Phase-2 follow-up. Until then, the approval signature *is real ECDSA-P256*, but it does not bind to a persistent reviewer identity across sessions or devices.
- **`/api/author/approve` is demo-grade auth.** UI-side passphrase only (`reviewer-alpha-42`); no server session, no CSRF, no rate-limiting. Production deployments must put this endpoint behind a real session bound to the reviewer's enrolled passkey before any classroom rollout. Captured in `SAFEGUARDING.md` §3.
- **Mock LLM provider is the default.** It produces clearly-labelled placeholder drafts that say `[MOCK DRAFT — REWRITE BEFORE APPROVAL]` in every field. This is by design; a silent fake would be worse than no draft at all. Set `LLM_PROVIDER=anthropic` (or `openai`) plus the corresponding API key env var to use a real provider.
- **Content breadth at v1.5.0: one fully-migrated, enriched maths skill family.** The 63 other tiles still need a teacher to draft + approve content. The pipeline does not write content; it distributes it.
- **Symbolic answer-checking shipped in v1.5.1.** The runtime answer-checker now handles both numeric and algebraic `expectedAnswer` values via math.js (MIT, native JS, ~600 KB, no Wasm). `simplify(actual - expected) == 0` decides polynomial / rational-function equivalence; a three-sample numeric fallback catches transcendental identities the symbolic simplifier doesn't fully reduce. Unicode superscripts and operators (`²`, `×`, `÷`, `−`) are normalised before parsing. Qualitative items still use the `qualitative-no-auto-check` sentinel and route to teacher-marking. 19 tests in `tests/unit/symbolic-checker.test.ts`. **Still not covered:** heavyweight graduate-level symbolic work (multivariable calculus, symbolic ODEs, abstract algebra) — that's the Pyodide+Sympy escalation path documented in `docs/ROADMAP_HIGHER_MATHS.md`. English short-answer / MFL spelling tolerance remains unimplemented.
- **LaTeX rendering shipped in v1.5.1.** KaTeX (MIT, ~280 KB) wraps via `lib/render/math.tsx`. Server-safe, `throwOnError: false`, `trust: false` (no raw `\\href` / `\\includegraphics`), degrades malformed input to an inline red marker rather than crashing the React tree. 6 tests in `tests/unit/math-render.test.ts`. Authored content can now include `\\frac`, `\\int`, `\\sum`, `\\begin{pmatrix}…\\end{pmatrix}`, etc., directly in `problem` / `explanation` / `workedSolution` fields.
- **Content manifest is served from `public/content/manifest.json`.** The browser registry (`lib/content/registry.ts`) verifies pack hashes and per-item ECDSA signatures against the manifest's trusted-reviewers list. If the manifest is missing or any verification fails, the registry returns an empty state and the engine falls back to the v1.4.5 hand-written `parallel-problems.ts` corpus. This is *additive by design* — the pipeline cannot regress the engine.
- **Difficulty-banded parallels are an authoring convention, not yet a schema field.** v1.5.0 introduces three bands for worked examples — `simpler` (drops one inverse operation, served first when a learner has failed all three Socratic hints), `same` (current behaviour, served after the warm-up), `stretch` (added structural step, served when the learner is asking for more). Today the band is encoded in the worked-example `id` prefix (`linear-eq-1var-simpler-NNN`, `linear-eq-1var-NNN`, `linear-eq-1var-stretch-NNN`); the engine's parallel-selector still serves examples in declaration order. **v1.5.1 will lift the convention into an explicit `band` field on the worked-example schema and teach the engine to honour it automatically.** The authoring intent is real and documented in-pack; the engine wiring is the half that remains. Demonstrated in `content/packs-raw/maths.linear-eq-1var.mjs:99-146`.
- **Single-reviewer signing is the v1.5.0 default.** Today one reviewer fingerprint is enough for an item to ship. v1.5.1 will require a primary plus a peer reviewer (two fingerprints) before a manifest accepts an item. Captured in `docs/WHY_NO_LLM_AT_RUNTIME.md` §2.2.
- **Per-item regression tests are not yet automated.** Every approved item *should* carry 2–3 sample (correct, wrong) answers regression-tested in CI; v1.5.0 ships without that, so a malicious or careless edit to an approved item could in principle reach the next manifest build. Mitigated structurally by the signature-invalidation property (any character change invalidates the signature and the registry rejects the item), so the worst case is *no content* served, never *wrong content* served. The v1.5.1 regression-test pack closes the gap entirely.

**See also.** The full conviction case for the no-LLM-at-runtime architecture and the honest accounting for content correctness now lives in `@c:\Users\Stu\New folder\docs\WHY_NO_LLM_AT_RUNTIME.md`. The open-source-only roadmap for higher-maths capabilities (KaTeX, math.js, JSXGraph, Pyodide+Sympy) lives in `@c:\Users\Stu\New folder\docs\ROADMAP_HIGHER_MATHS.md`. Both documents exist so future proposals reference one source of truth rather than re-litigate the architecture each time.

### 4.4 Higher-maths capability stack (v1.5.1 + v1.5.2 + v1.5.3)

**Update for v1.5.3.** The three "phase 2" gaps disclosed in v1.5.2
are now closed:

- *Heavy CAS now used by an answer-checker dispatcher.*
  `lib/validation/answer-checker-heavy.ts` exposes `diagnoseHeavy(...)`,
  an async wrapper that runs math.js first, then escalates to Pyodide
  +Sympy ONLY when math.js says `wrong` for a symbolic answer. The
  ENGINE still calls the synchronous `diagnose(...)` (no model in the
  hot path); surfaces that explicitly want the heavy escalation
  import the new module. Always degrades silently to the math.js
  verdict on any CAS failure (unavailable, timeout, abort, error).
  Pinned by 10 protocol tests using a stub CAS client.

- *First authored figure now shipping.* The `maths.linear-eq-1var`
  pack item `001` carries a `figures` entry: a graph of `y = 2x + 5`
  (the LHS of the problem). The figure deliberately omits the
  `y = 17` reference line and the intersection marker — graph-based
  estimation is permitted as a cross-check method, but the answer
  isn't decorated onto the figure. The schema gained an optional
  `figures?: readonly FigureSpec[]` field; the validator at load
  time runs each through `validateFigureSpec` and rejects the item
  if any is malformed. The build script rebuilds + re-signs the
  pack with the figure present.

- *Move-vocabulary scope expanded.* The v1.5.2 disclaimer that
  "`square both sides`, `take log`, `apply sin to both sides`,
  variable-operand moves are out of scope" is no longer accurate.
  v1.5.3 ships:
    - Variable-operand arithmetic (`multiply by x`, `divide by (x+1)`).
    - `square both sides` as its own move kind.
    - `apply f` for a whitelisted set of unary functions (sin, cos,
      tan, asin, acos, atan, sinh, cosh, tanh, exp, log, log10,
      log2, ln, sqrt, cbrt, abs). `ln` is normalised to `log` because
      math.js's `log(x)` is the natural log.
  The `verifyMove` algorithm was rewritten: instead of a kind-keyed
  ratio check on diff forms, it now CONSTRUCTS the expected next
  equation from the previous one and the move, then compares strictly
  (no scaling) against the learner's actual next equation. This
  unifies all move kinds. Pinned by 37 tests.

What's STILL out of scope (deferred to v1.5.4+):

- **Substitution moves** — `let u = x²`, `let t = sin(x)`, etc. These
  introduce a new variable, which `verifyMove` doesn't handle. The
  validator returns `parse-error` for attempts.
- **Case-analysis moves** — `case 1: x ≥ 0; case 2: x < 0` and the
  branched continuations. Out of scope; the step-validator only
  walks a single linear chain of equations.
- **Integration / differentiation embedded in a step** — e.g. a
  derivation that takes the derivative of both sides. The `apply-fn`
  whitelist excludes `diff` and `int` deliberately, because a sample-
  point check can't verify them (you'd need symbolic equality).
- **Conditional rewrites** — `divide by (x − 1)` is technically only
  valid where `x ≠ 1`; `verifyMove` doesn't track domain restrictions.
  A learner doing this gets `ok` even though spurious solutions may
  later appear. Documented at the top of the move-vocabulary file.

### 4.4-historical Higher-maths capability stack (v1.5.1 + v1.5.2)

The proposal-for-Dowling and the higher-maths roadmap committed to
closing three gaps in the maths-capability matrix. v1.5.1 delivered
symbolic equivalence, KaTeX rendering, and the basic step validator;
v1.5.2 delivers the heavy CAS, geometric figures, and move-vocabulary
verification. What follows is the honest accounting.

**Heavy CAS (Pyodide + Sympy).** `lib/cas/heavy-client.ts` and
`public/workers/heavy-cas.worker.js` ship a real, working web-worker
wrapping Pyodide and Sympy. Operations supported: `simplify`,
`expand`, `factor`, `integrate` (indefinite), `diff` (any order),
`solve`, `ode`, `version`. Each returns both string and LaTeX form.
What this is and isn't:

- **It IS opt-in.** No engine code path autoloads the worker.
  Surfaces opt in by importing `runCAS(...)` or `getHeavyCAS()`.
- **It DOES make a network request.** Pyodide assets (~10-20 MB
  WASM blob, plus sympy package) are fetched from jsDelivr by
  default on first use. This is the *only* outbound traffic the
  CAS worker generates. Learner text never leaves the device —
  Pyodide is a local interpreter, not a remote API.
- **It IS self-hostable.** Pass `{ indexURL: "/pyodide/" }` to
  `getHeavyCAS()` and serve the pinned Pyodide release tarball
  from `public/pyodide/`. Same for the worker URL itself.
- **It is NOT validated against real Pyodide in the unit suite.**
  Pyodide can't run in Node — it loads WASM via `importScripts`.
  The 9 unit tests pin the request/response *protocol* using a
  mock worker. End-to-end validation against real Pyodide is in
  the Playwright suite (E2E follow-up).
- **It is NOT used by the engine yet.** The engine still calls the
  math.js path (`diagnose` dispatcher). Surfaces that want heavy
  CAS for authoring or for advanced calculus problems must wire
  it in explicitly. Phase 2: a `diagnoseHeavy(...)` dispatcher
  that calls `runCAS(...)` when math.js returns inconclusive.

**Geometric figures (JSXGraph).** `lib/geometry/figure-spec.ts`
defines a typed JSON schema for figures that can ride inside signed
content packs. `components/shared/GeometryFigure.tsx` lazy-loads
JSXGraph from a CDN at first mount and renders the spec. What this
is and isn't:

- **The spec validator is pure.** `validateFigureSpec` runs in
  Node, in the build script, and in the registry. Malformed
  figures cannot ship.
- **The renderer is browser-only.** SSR yields a placeholder.
  When JSXGraph fails to load (offline / blocked CDN), the
  placeholder includes the figure's `alt` text so screen-reader
  users still get a description.
- **It does NOT yet have authored figures.** No content pack ships
  with a figure today. The schema is in place, the renderer is in
  place, the build script accepts figures — what's missing is
  authored content from a reviewer. Phase 2.
- **It does NOT cover everything.** Conics by general equation,
  parametric curves, animated sliders, 3D, hyperbolic geometry —
  all out of scope for v1.5.2. The matrix in
  `docs/ROADMAP_HIGHER_MATHS.md` tracks them.

**Step move-vocabulary (`lib/validation/move-vocabulary.ts`).** The
v1.5.1 step validator caught arithmetic slips and sign flips; v1.5.2
also catches the case where the learner *says* one thing and *does*
another. What this is and isn't:

- **It IS narrow.** Only numeric-operand additive / multiplicative
  moves and the no-operand expand / factor / simplify moves are
  recognised. `square both sides`, `take log`, `apply sin to both
  sides`, substitution moves with a variable operand — all out of
  scope for v1.5.2. Disclosed in the in-file header.
- **It IS a soft signal, not a hard fail.** A move-mismatch flags
  a vocabulary slip; the working may still be algebraically
  correct. The validator surfaces `move-mismatch` distinct from
  `invalid` so engines can route them differently.
- **It DOES preserve the leakage discipline.** The hint never
  echoes the right operand or the right answer. Pinned by tests.

### 4.5 Hallucinated or confused names

- `lib/types/index.ts` line 1 comment says **"Core Data Models for Luminary x VertoLearn"** — "Luminary" is a previous product name that has been scrubbed elsewhere. Cosmetic only. Will clean up.
- `IndexedDB` schema name is `"LuminaryOfflineDB"` in `hardware-ghost.ts`. Renaming would force a schema migration for anyone who already has local data — left as-is.

### 4.6 Security honesty

- **Privileged surfaces are passphrase-gated, not authenticated.** `RoleGuard` on `/teacher` and `/compliance` requires a shared passphrase (demo defaults `mentor-alpha-42` / `officer-alpha-42`); a child who learns the passphrase from a teacher can still access those surfaces. There is no per-user audit trail. Phase 2 → WebAuthn passkeys. See SAFEGUARDING.md §3.
- **Age band is self-declared.** A child can pick `18-plus` to bypass the under-13 guardian-acknowledgement step. This does not satisfy COPPA §312.5. Phase 2 → verified parental consent (email-OTP / KYC / school roster). See SAFEGUARDING.md §4.
- **No server-side validation of anything.** There is no server.
- **Session keys are not stored; passkeys are.** The session-demo ECDSA keypair is regenerated per page load — signatures verify within a session, nothing cross-session or cross-device. v1.4.11 adds an **optional WebAuthn passkey** the learner can enrol on `/student`; the enrolment record (`credentialIdB64url`, `publicKeyB64url`) persists in `localStorage` and the private key lives in the authenticator (TPM / Secure Enclave / security key). Receipts signed via passkey carry `keyType: "passkey-derived"` and are verifiable across sessions on the same device / synced-passkey cluster. Receipts signed via the session key remain labelled as such and are explicitly **not** identity-bound. Neither path involves a server.
- **`ZK_SALT` defaults to `"default-salt-change-in-production"`** in `aggregator.ts`. If this code ever runs server-side, this must come from env.
- **`crypto-js` is an older library.** Fine for SHA-256 hashing but its AES implementations have known ECB/padding pitfalls. We use it only for hashing.

## 7. Principles that are actually enforced by code

These are not marketing; they are enforced by code paths that exist today:

- **No direct answers from Eke.** Enforced because there is no LLM at all — only `lib/eke/tiered-hints.ts` templates with the `hintContainsAnswer()` validator. You cannot accidentally ship an answer because one cannot be computed.
- **No biometrics.** Enforced by the absence of any biometric API call. No `mediaDevices`, no `navigator.credentials.get({publicKey: {userVerification: "required"}})`. Grep for it — you will find nothing.
- **No advertising.** Enforced by the absence of any ad network script.
- **No tracking beacons.** Enforced by the absence of any analytics library.
- **Zero-paste default.** Enforced by `EkeChat.tsx` `onPaste={(e) => e.preventDefault()}` when `zeroPaste=true` (the default).
- **Crisis-language handoff.** Enforced by `decision-gate.ts` crisis signals — returns a Childline-equivalent message **before** any other processing.
